Field Packet

The Field Packet

A plain-language walk-through of the corridor ceremony — getting the folder onto each machine, checking the machine is healthy, and running the conversation that proves two computers can talk to each other directly, with no server and no account in between.

Chapter One

Before the call

Before anyone picks up the phone, the same folder needs to exist on both computers — one in Bahrain, one in India. How it gets there matters more than it looks like it should.

Prefer this  Use a USB stick if you possibly can

Windows attaches an invisible "mark of the web" to anything that arrives over the internet, and it treats that mark with suspicion. A copy made from a USB stick or over a local network usually carries no such mark and opens silently — no prompt at all.

If it has to travel over the internet instead

A folder downloaded from a browser, or unzipped from a downloaded file, will very likely show a blue screen the first time the program is opened, reading "Windows protected your PC." This is not a sign that anything is wrong. The way through it is:

More info  →  Run anyway

An emailed zip file behaves the same way, and some mail programs strip or block the program file entirely before it even arrives — best avoided if there is any other way to get the folder across.

Honesty note

This kit is not digitally signed by a publisher — that was a deliberate choice, not an oversight. A warning on a downloaded copy is therefore expected behaviour, not evidence that something has gone wrong. Never promise "you won't see any warnings." Instead, tell the person which warning they may see, and exactly which two buttons close it.

If the person on the other end isn't comfortable clicking through a security prompt on their own, the right answer is to hand them the folder on physical media instead. That's a judgment call about the person, not a technical problem to solve.

Matters more than it looks  Where the folder lives

The folder's location on the machine must not contain a # character anywhere in its path. This is a real, known quirk in the ceremony software, and a folder placed somewhere like a Windows username containing a # will fail in a way that looks exactly like a broken download — confusing, and hard to diagnose over the phone.

Put the folder somewhere short and boring: C:\corridor\ or straight onto the Desktop. The kit's own health check (Chapter Two) looks for this problem and warns about it before anyone's time is wasted — run that first if there's any doubt.

Synthetic data only

Every name and every message used while running the ceremony must be made up — a made-up person, a made-up message. No real customer, no real document, and no real business information, ever, including in the test message both sides type to prove it works. This rule holds every time the kit is run, not just on the day that matters.

Chapter Two

The health check

Before the ceremony call, double-click verify_clean_machine.cmd inside the folder. It checks the machine on its own, with nobody else needing to be on the line, and it proves whether this particular computer can hold up its end of the conversation.

What a healthy machine says

Here is the exact screen from a machine that passed — this is what "good" looks like:

A: ok, not resolvable: node / npm / npx / bare
A: MACHINE IS CLEAN for the Node-free claim.
A: informational: vcruntime140.dll in System32 = False
C: probe control correctly NOT ok (verdict=CONTENT_FAIL)
D: TALLY  OK=16/16  HANG=0/16  CONTENT_FAIL=0/16
=== VERDICT: PASS -- CLEAN machine, 16/16 content-verified ceremonies. ===

Verbatim console output from a passing run, Windows Sandbox, 2026-07-20.

Sixteen practice conversations, run silently, all sixteen came back readable. That line reading VERDICT: PASS is the only line that matters — everything above it is the detail behind that verdict.

What to send back

Whatever the result — pass or otherwise — send back the whole VERIFY_EVIDENCE.txt file together with the whole verify-logs\ folder next to it. That pair is what makes it possible to work out what happened without needing a second call.

Field notes — things that look wrong but aren't

The first run may be noticeably slower than the rest. Windows Defender scans an unfamiliar 45-megabyte program the first time it runs. That pause is normal and only happens once — it is not a failure.

If a window flashes open and disappears before anything useful appears on screen, that is a real and diagnosable event — but not one to keep retrying blind. Photograph what appeared, however brief, and report it rather than clicking through it again and again.

Chapter Three

The conversation

This is the ceremony itself — two people, one phone call. Before anything else, decide out loud which of you is starting the conversation and which of you is joining it. It does not matter which machine plays which role, but the decision needs to be made before you begin, not worked out halfway through.

Both people, at the same time: double-click run_bare_mesh.cmd, then type 2 and press Enter, for "Open the messenger." Windows will show a one-time permission notice to each of you — press Enter, or type skip and press Enter. Either is fine. You will then both see the same question on screen, and this is where your two paths split.

Role

The person STARTING

  1. 1Type connect and press Enter.
  2. 2At the question "Did someone send you a code?" — just press Enter. You are the one starting, so there is no code yet.
  3. 3The screen prints a long code. Send it — by WhatsApp, email, whatever is easiest to hand. Reading it aloud is the fallback, not the plan; if it has to be read aloud, it is printed in groups of four to make that easier.
  4. 4The screen then waits and says it is listening. Leave this window open.
  5. 5The other person will read you back a short code. Type or paste it in and press Enter. The screen then says "the other computer can finish joining now."
  6. 6You are now in the messenger. Type a message and press Enter. Type /read to see the conversation.

Role

The person JOINING

  1. 1Type connect and press Enter.
  2. 2Paste the long code they sent you and press Enter. Extra spaces or line breaks picked up from a chat app are fine — they are stripped out automatically.
  3. 3Your screen prints a short code. Read it back to them out loud — this one is short enough to say, in groups of four.
  4. 4You will be asked for an address. If you are in different offices or different countries, just press Enter. Only type something here if you are both on the very same office network and someone has told you an address such as 192.168.1.5:4300.
  5. 5The screen says it is waiting for the other computer to let you in. This can take a little while. When it succeeds, it says "joined — you can post now."
  6. 6Type a message and press Enter. Type /read to see the conversation.

When you are both done, type /exit and then 5 to close.

Proving it actually worked

Do both of the following, in this order. One direction on its own is not a corridor:

  1. The person who joined types a message. The person who started types /read and reads it aloud.
  2. The person who started types a message. The person who joined types /read and reads it aloud.
Use /read, never /rooms

Type /read, not /rooms. /rooms shows only a one-line summary of the last message in the room's own internal order — which is not necessarily the newest message just received. If both people type at close to the same moment, a message that arrived perfectly can still be missing from that summary line. /read shows the real conversation. This isn't a small nicety — a working corridor was once wrongly declared broken for exactly this reason.

What to photograph

  • Both screens at the moment of the verdict — the joiner's "joined — you can post now" and the starter's "the other computer can finish joining now."
  • Both /read outputs from the proof above, showing each side's message on the other's screen. This is the actual proof; everything else is supporting evidence.
  • Any red or unexpected line, in full — including the detail printed underneath it. The screen deliberately prints one plain sentence first and the raw detail below; photograph both parts, not just the sentence.
  • If anything failed along the way: the whole VERIFY_EVIDENCE.txt and verify-logs\ folder from Chapter Two.
Never photograph

Never photograph the long invite code. It is a bearer secret — whoever holds it can read the room. Send it once, directly to the one person who needs it, and never place it in a group chat or a support ticket.

What "it worked" actually means

None of the following count as proof, on their own:

  • Not proof  A window that closed without showing an error. This software can report success even on a real failure, including a total loss of its own output — the closing behaviour alone tells you nothing.
  • Not proof  "It says Goodbye." A kit that ran its whole ceremony and silently failed to post anything will still say Goodbye at the end.

The only proof  is content, in both directions, in the same session: a message typed on machine A is visible on machine B, and a message typed on machine B is visible on machine A, with the actual words matching. Anything less than both of those is not a corridor.

Before you escalate

If something goes wrong

Run verify_clean_machine.cmd again and send back the whole evidence folder — VERIFY_EVIDENCE.txt and verify-logs\ together, not just one or the other.

A single red connection check is not proof that anything is broken. In measurement, roughly one check in seven came back red on a connection that was in fact working fine, against a verified control that confirmed the measurement itself was real. Try the check again before treating it as bad news and escalating.
