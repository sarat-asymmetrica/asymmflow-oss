# Asymmetrica Sovereign Software Constitution

Status: Draft v0.1
Created: 2026-05-22
Scope: AsymmFlow 2026 Edition and reusable Asymmetrica operational software

## Purpose

This document defines the north star for AsymmFlow's pre-release architecture.

The goal is no longer to complete a vertical ERP suite, accumulate SaaS
features, or imitate enterprise procurement theater. The goal is to build
sovereign operational software:

```text
working system
+ source code
+ schemas
+ invariant rules
+ harnesses
+ migration methodology
+ recommended AI coding workflows
+ capability maps
```

The customer owns their downstream operational reality. Asymmetrica stewards the
canonical upstream.

## Core Promise

```text
Software you can truly own.
```

Ownership means the user can:

- run the system locally or in infrastructure they control;
- understand the operational shape of the system;
- inspect source code, schemas, migrations, and harnesses;
- adapt workflows without losing continuity;
- use AI to reduce maintenance burden without surrendering authority;
- consult Asymmetrica upstream maintainers when deeper support is useful;
- avoid forced subscription dependency and roadmap churn.

This is ownership-first, not anti-cloud. Managed trust infrastructure is allowed
when it improves reliability, security, or implementation fit. It must remain an
explicit trade-off, not an invisible lock-in.

## Human Philosophy

The software should express:

- respect;
- clarity;
- ownership;
- agency;
- operational dignity;
- human-scale trust;
- local-first continuity;
- transparent trade-offs.

The system should make complexity navigable without making operators feel small.
It should help people grow in capability, not dependency.

## Business Model

The primary model is edition-based sovereign software.

An edition, such as `AsymmFlow 2026 Edition`, includes:

- runnable software;
- source code;
- deployment stack;
- schemas;
- harnesses;
- migration tooling;
- capability docs;
- AI integration methodology;
- implementation fit guidance;
- upstream support boundaries.

After purchase or adoption, the customer owns a sovereign fork. Asymmetrica
maintains the canonical upstream.

## Release Philosophy

Do not release on a forced annual treadmill.

Release a new edition only when meaningful evolutionary change has occurred:

```text
2026: major architectural leap -> edition release
2027: no major evolution -> maintenance/security only
2028: meaningful AI/runtime/schema evolution -> edition release
```

Avoid:

- feature churn;
- roadmap pressure;
- subscription theater;
- forced upgrade anxiety;
- minor version marketing disguised as progress.

## Upstream And Downstream

### Canonical Upstream

Asymmetrica maintains:

- primitive kernel;
- reusable engines;
- schemas;
- canonical workflows;
- migration rules;
- invariant suites;
- reference implementations;
- capability docs;
- harnesses;
- security advisories;
- core bug fixes.

### Sovereign Downstream

Customers own:

- local workflows;
- local integrations;
- local modifications;
- domain-specific automations;
- operational data;
- deployment decisions;
- custom overlays and fork-specific rules.

Asymmetrica does not become the permanent maintainer of every customer fork.
That boundary is central to the model.

## Support Philosophy

Included canonical stewardship:

- security advisories;
- upstream bug fixes;
- schema guidance;
- migration guidance;
- critical upstream support;
- reference harness updates.

Separate paid work:

- fork debugging;
- custom integrations;
- custom workflows;
- customer-specific feature additions;
- deep architecture support;
- managed migration projects;
- hosted or managed operations.

No support promise may imply infinite maintenance of arbitrary downstream forks.

## AI-Assisted Evolution

AI makes this model viable because it reduces the cost of:

- codebase understanding;
- integration work;
- migration work;
- fixture generation;
- debugging;
- repair;
- documentation;
- local customization.

AI may inspect, explain, classify, draft, recommend, scaffold, and propose
patches. AI must not silently mutate authoritative operational state.

The authority loop is:

```text
AI proposes
-> deterministic layer validates
-> invariant harnesses run
-> human approves
-> system executes
-> audit trail records
```

## Architectural Direction

The pre-release objective is to transform the current industrial/trading ERP
line into the canonical sovereign operational substrate.

The stable architecture is:

```text
Primitive Kernel
-> Operational Engines
-> Domain Packs / Overlays
-> Sovereign Forks
```

The current PH/trading/instrumentation shape is not discarded. It becomes the
first serious domain overlay and proof that the substrate can support real
business complexity.

## Non-Goals

This project is not trying to become:

- extractive SaaS machinery;
- a cloud-only app;
- a low-code form builder with weak invariants;
- an enterprise certification performance before demand exists;
- a support trap where every customer fork becomes upstream responsibility;
- a pile of screens without owned operational meaning;
- an AI agent that bypasses human authority.

## Rigor Commitments

Every strategic claim must have an epistemic status:

- proven in code/tests;
- verified by harness;
- production-used;
- pilot-ready;
- prototype;
- research;
- assumption-scaffolded;
- speculative.

Every major capability must state:

- problem solved;
- inputs;
- outputs;
- human approval boundary;
- invariant and harness strategy;
- best-fit use cases;
- known limitations;
- implementation trade-offs;
- maturity classification.

## Current Proof Loops

The first proof loops are:

- Business Memory Intake: source evidence to reviewable candidate to context pack
  and audit/export provenance.
- Cashflow Evidence: invoices, payments, bank/document facts, posting coverage,
  trial-balance gates, allocation evidence, action proposals, and export packs.
- Accounting Posting Spine: deterministic posting previews, draft journals,
  coverage report, and review-first accounting gates.
- Source Asset Registry: durable provenance for files, inbox records, scans,
  emails, folders, and future source-bearing overlays.

These loops already express the sovereign model: evidence is preserved, AI is
bounded, deterministic services retain authority, and operators can inspect,
correct, export, and approve.
